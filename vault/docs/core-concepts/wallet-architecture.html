Wallet Architecture

This document describes how Jutis is built — the non-custodial security model, how private keys are handled locally, and how the extension communicates with web applications and the Canton network.

Non-custodial model

Jutis is a non-custodial wallet. This means the private keys that control your Canton assets are generated and stored exclusively on your local device. Jutis never has access to, copies of, orbackup of your private keys at any point.

In a custodial model (like a centralized exchange), the institution holds your private keys and can freeze or move your assets. In Jutis, only you hold the keys. The extension is simply an interface — a way to interact with the Canton network without exposing sensitive data.

info

Recovery phrase consequence

Because Jutis is non-custodial, if you lose your recovery phrase and your extension data, there is no recovery path. Jutis support cannot recover your keys, assets, or account. This is by design — not a limitation.

Local key handling

When you create a new wallet in Jutis, a cryptographic key pair is generated using industry-standard algorithms (Ed25519 or similar, depending on Canton requirements) directly inside the browser extension's isolated storage.

The key handling lifecycle:

  1. Generation — Keys are generated inside the extension sandbox. The raw private key never enters the JavaScript heap in plaintext.
  2. Storage — Private keys are encrypted with your local wallet password and stored in the extension's local storage area. They are never written to disk unencrypted.
  3. Signing — Signing operations (message signatures, transaction authorization) happen inside the extension. The private key is decrypted momentarily for the operation, then re-encrypted immediately.
  4. Export — The only way to export key material from Jutis is via the 12 or 24-word recovery phrase, which you control entirely.

Extension architecture

The Jutis browser extension consists of a background service worker and a content script that runs on pages you authorize. This architecture separates wallet operations from the web pages you visit.

Background service worker

Handles key storage, signing operations, and Canton network communication. The content script cannot access private keys directly.

Content script (provider)

Injects the window.jutis provider into web pages. Handles connection requests, emits events, and proxies signing requests to the background worker.

This means a malicious or compromised web page cannot access your private keys even if it can detect the Jutis extension. All sensitive operations are gated through the extension's confirmation UI.

App connection flow

When a web application (dApp) wants to connect to Jutis, it calls window.jutis.requestAccounts(). Here's what happens step by step:

  1. The dApp calls jutis.requestAccounts() from its own page JavaScript
  2. The content script proxies this request to the background service worker
  3. The extension displays a confirmation popup asking the user to approve or reject
  4. If approved, the user's Canton address is returned to the dApp via the Promise
  5. If rejected, the Promise rejects with ACTION_REJECTED

The dApp receives only the address — never the private key, recovery phrase, or any other sensitive data. This approval is per-session per-site and can be revoked at any time from the extension settings.

Signing approval flow

When a dApp requests a signature (e.g., to authenticate a user or authorize a Canton transaction), the same confirmation gate applies:

  1. The dApp calls signer.signMessage(message) or initiates a transaction
  2. The extension shows a popup with the full message content or transaction details
  3. The user reviews the content and clicks Approve or Reject
  4. Only on approval does the signing operation execute locally and return the signature
  5. Jutis never signs anything the user has not explicitly reviewed and approved
warning

Always verify what you're signing

Review the full message or transaction details in the Jutis popup before approving. Never approve signing requests for messages you didn't initiate or transactions you don't recognize.

Security boundaries

Jutis enforces strict boundaries between the extension and the outside world:

  • No key export without user action — Recovery phrase is only shown on explicit user request, not accessible programmatically
  • Content script isolation — The injected provider cannot read extension storage or trigger signing without user approval
  • No network access from signing module — Signing operations execute in an isolated context with no outbound network calls
  • SOC 2 Type II audit in progress — External audit verifies these boundaries are enforced correctly
  • Pen testing completed Q1 2026 — Third-party testing identified no critical vulnerabilities in the extension boundary model
